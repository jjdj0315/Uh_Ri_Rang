# DAKER 서비스 구조 설명서

## 1. 서비스 한줄 요약

DAKER는 **해커톤 플랫폼**이다. 해커톤 목록 조회, 팀 모집, 제출, 리더보드, 랭킹을 제공하며, AI가 팀을 매칭해주는 확장 기능을 포함한다.

---

## 2. 기술 스택

```
Next.js (App Router)  ─  React 기반 풀스택 프레임워크
Tailwind CSS          ─  유틸리티 기반 스타일링
shadcn/ui             ─  UI 컴포넌트 라이브러리
next-themes           ─  다크 모드
localStorage          ─  클라이언트 데이터 저장 (백엔드 없음)
Gemini API            ─  AI 팀매칭 (Next.js API Route 경유)
Vercel                ─  배포
```

## 4. 데이터 흐름

### 백엔드가 없다 — localStorage가 DB 역할

```
앱 최초 로드
    │
    ▼
localStorage에 데이터가 있는가?
    │
    ├─ NO  → 예시 JSON 파일(data/예시자료/*.json)을 localStorage에 초기화
    │
    └─ YES → localStorage에서 읽어서 렌더링
```

### 데이터 4종류

| Key            | 내용             | 읽기 페이지                  | 쓰기 페이지           |
|----------------|-----------------|-----------------------------|-----------------------|
| `hackathons`   | 해커톤 목록+상세  | `/`, `/hackathons`, 상세     | (초기화만)             |
| `teams`        | 팀 목록          | `/camp`, 상세의 Teams 탭     | `/camp` (팀 생성)      |
| `submissions`  | 제출물           | 상세의 Leaderboard 탭        | 상세의 Submit 탭       |
| `leaderboards` | 리더보드 점수     | 상세의 Leaderboard, `/rankings` | Submit 시 자동 갱신  |

```
┌─────────────────────────────────┐
```

```
├─────────────────────────────────┤
│  ┌──────┐  ┌──────┐  ┌──────┐  │
│  │카드1  │  │카드2  │  │카드3  │  │
│  │제목   │  │제목   │  │제목   │  │
│  │상태   │  │상태   │  │상태   │  │
│  │태그들 │  │태그들 │  │태그들 │  │
│  │기간   │  │기간   │  │기간   │  │
```

동작:

2. 필터 조건에 따라 목록 필터링 (status, tags)
3. 카드 클릭 시 해당 slug의 상세 페이지로 이동
4. 결과 없으면 빈 상태 UI 표시

### 5-3. 해커톤 상세 (`/hackathons/:slug`)

```
┌─────────────────────────────────────────────┐
│  해커톤 제목                                  │
├─────────────────────────────────────────────┤
│  [개요] [안내] [평가] [일정] [상금] [Teams]   │
│  [Submit] [Leaderboard]                      │
├─────────────────────────────────────────────┤
│                                             │
│         선택된 탭의 콘텐츠 표시                │
│                                             │
└─────────────────────────────────────────────┘
```

핵심: **하나의 페이지, 탭으로 섹션 전환**

| 탭 | 데이터 소스 | 주요 동작 |
|----|-----------|----------|
| 개요 | hackathon_detail.sections.overview | 읽기만 |
| 안내 | sections.info | 읽기만 |
| 평가 | sections.eval | 읽기만 |
| 일정 | sections.schedule | milestones 타임라인 |
| 상금 | sections.prize | 순위별 상금 표시 |
| Teams | localStorage `teams` (hackathonSlug 필터) | 팀 목록 + `/camp`으로 이동 |
| Submit | sections.submit | **폼 입력 → localStorage 저장** |
| Leaderboard | localStorage `leaderboards` | 순위 테이블, 빈 상태 처리 |

## 6. AI 팀매칭 (Track 1) 동작 흐름

```
사용자                     /camp 페이지                 /api/match              Gemini API
  │                           │                           │                       │
  │ "React 프론트 개발자       │                           │                       │
  │  있는 팀 찾아줘"           │                           │                       │
  │ ──────────────────────▶  │                           │                       │
  │                          │  POST /api/match           │                       │
  │                          │  { query, teams[] }        │                       │
  │                          │ ─────────────────────────▶ │                       │
  │                          │                           │  Gemini에 프롬프트 전송 │
  │                          │                           │  "다음 팀 목록에서      │
  │                          │                           │   사용자 조건에 맞는    │
  │                          │                           │   팀을 추천하고        │
  │                          │                           │   근거를 설명해줘"     │
  │                          │                           │ ─────────────────────▶│
  │                          │                           │                       │
  │                          │                           │  ◀─ 추천 결과 JSON ───│
  │                          │  ◀─ { matches, reasons } ─│                       │
  │  ◀── 추천 팀 카드 + 근거 ─│                           │                       │
  │      표시                 │                           │                       │
```

### API Route 구조 (`/api/match`)

```
요청: POST /api/match
Body: {
  "query": "React 프론트 개발자 있는 팀",
  "hackathonSlug": "aimers-8-model-lite"  // optional
}

응답: {
  "matches": [
    {
      "teamCode": "T-BETA",
      "teamName": "PromptRunners",
      "reason": "Frontend 개발자를 모집 중이며, 1명으로 구성되어 합류 여지가 큽니다.",
      "score": 0.92
    }
  ]
}
```

핵심: **Gemini API 키는 서버(API Route)에서만 사용** → 클라이언트에 노출되지 않음.

## 8. 핵심 정리

| 항목 | 설명 |
|------|------|
| 백엔드 | **없음**. localStorage가 DB 역할 |
| 서버 코드 | `/api/match` (Gemini 호출)만 유일한 서버 로직 |
| 데이터 초기화 | 앱 최초 로드 시 예시 JSON → localStorage 복사 |
| 상태 관리 | React state + localStorage (별도 상태 라이브러리 없음) |
| 배포 | Vercel (정적 + API Route 서버리스) |
| AI 기능 | 팀 매칭 검색 — 자연어 → Gemini → 추천 결과 |
